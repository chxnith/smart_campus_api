### Question 2: Why is the provision of ”Hypermedia” (links and navigation within responses) considered a hallmark of advanced RESTful design (HATEOAS)? How does this approach benefit client developers compared to static documentation?

-"HATEOAS" (hypermedia) is often regarded as a defining feature of advanced RESTful services since the server responds not only with the resource representation but also with navigation instructions that allow for driving the client application state. This allows for a higher level of self documentation and makes the API work similar to the Web ("the client follows the links") where the developer does not need to have any knowledge about the URI structure. For instance, the client could navigate through self, rooms, sensors, and readings dynamically based on the response rather than knowing about the available resources beforehand and crafting a URI accordingly. Thus, the client becomes decoupled from the URI structure and is more robust against changes in the API due to evolving paths without changing the links.


### Question 3: When returning a list of rooms, what are the implications of returning only IDs versus returning the full room objects? Consider network bandwidth and client side processing.

-Sending back only IDs for rooms keeps the size of the list response small, which in turn conserves bandwidth, minimizes latency, and is highly efficient if the client requires only references (eg: populating a drop-down menu, or fetching the details later upon request); on the flip side, the client would have to make more requests for names/ capacities, making the process more costly and time-consuming. In contrast, sending back full objects for the list response delivers a richer content in one shot, streamlining the client code and reducing follow-up requests (useful if the client UI needs to present room information right away), although at the cost of higher message payload size and higher processing overheads for the client, especially when dealing with a large number of objects with nested attributes. The most popular approach to handle this problem is to send a “summarized object” (eg: ID, name, possibly capacity) back for the list request, reserving detailed objects for the GET / rooms/ {id} case.


### Question 4: Is the DELETE operation idempotent in your implementation? Provide a detailed justification by describing what happens if a client mistakenly sends the exact same DELETE request for a room multiple times.

-Yes, the DELETE is idempotent regarding server state in this design.
The following occurs when the client sends DELETE / rooms/ {roomId} for the first time: either the room will be removed, along with all the sensors/readings related to it, provided that the room exists and is ready for removal, and 204 No Content will be returned or the room will not be removed because it still has sensors other than OFFLINE, even though the room exists, and 409 Conflict will be returned without altering any states. In case the client mistakenly repeats the identical DELETE after successfully deleting the room, there is no second removal of the room because the room has already been deleted, and thus the server responds with 404 Not Found without further changes. More significantly, despite different status codes being returned (204 followed by 404), there is no change in the state of the system whether the DELETE is executed for the first or tenth time, the room and all the data associated with it remain deleted.


### Question 5: We explicitly use the @Consumes (MediaType.APPLICATION_JSON) annotation on the POST method. Explain the technical consequences if a client attempts to send data in a different format, such as text/plain or application/xml. How does JAX-RS handle this mismatch?

-The Consumes(MediaType.APPLICATION_JSON) indicates to the JAX-RS engine that the POST operation supports the media types of the incoming payload with JSON (application/ json) content type only. Suppose a client sends data to the same endpoint but uses some other media types such as text and application/ xml. In that case, the JAX-RS engine will first try to determine if the POST method is capable of “consuming” the received content types. Since the POST method does not support consuming text/ plain or application/xml content types, the request will be immediately rejected. Typically, the client will receive an HTTP 415 Unsupported Media Type response (or some related client error response code) from the server because the JAX-RS engine could not find a compatible MessageBodyReader to process that media type for conversion to the target Java parameter type (Room or Sensor).


### Question 6: You implemented this filtering using @QueryParam. Contrast this with an alterna-tive design where the type is part of the URL path (e.g., /api/vl/sensors/type/CO2). Why is the query parameter approach generally considered superior for filtering and searching collections?

-When @QueryParam is used with the GET /sensors?type=CO2 request, “type” is seen as an optional filter applied to the resource collection, which is very much in line with the REST way of thinking, where the path specifies the resource (the sensors collection in /sensors), while the query string is used to specify filters/ search criteria/paging/sorting of the resource representation. On the other hand, the alternative approach of specifying the filter using the path itself (/sensors/type/CO2) suggests that the filter is another resource, which leads to lots of different endpoints (eg: /type/CO2/status/ACTIVE/room/ENG-110). This approach looks quite cumbersome to use when the filters are optional and can be combined.


### Question 7: Discuss the architectural benefits of the Sub-Resource Locator pattern. How does delegating logic to separate classes help manage complexity in large APIs compared to defining every nested path (e.g., sensors/{id}/readings/{rid}) in one massive con-troller class?

-However, the Sub Resource Locator design pattern brings substantial architectural value as it allows delegating a sub-domain of URI space for a top-level resource to a separate class (for example, SensorResource delegates /sensors/{id}/readings to SensorReadingResource). Such an approach contributes to better separation of concerns, dividing the responsibility of performing actions related to sensors (creating, listing, and retrieving them) from actions regarding their readings (listing and adding). As a result, both classes will be easier to implement, test, and maintain. Another advantage of using sub-resources is scalability since developers don’t need to manage one huge controller with all nested paths and every possible combination of path parameters; instead, the logic for each sub-resource can reside within that particular sub resource and won’t influence other classes. Lastly, sub-resources provide more flexibility for extending the API in the future without changing the parent resource, meaning that, for instance, /sensors/{id}/readings/stats could be implemented independently of the rest of the API.


### Question 8: Why is HTTP 422 often considered more semantically accurate than a standard 404 when the issue is a missing reference inside a valid JSON payload?

-HTTP 422 Unprocessable Entity is often a more semantic choice than 404 Not Found if the client makes a well-formed, proper JSON request on a valid route but references a value that cannot be resolved by the server (for example, roomId refers to a non-existent room). In this case, nothing seems to be "missing" from the request since the client is sending an existing route (POST /sensors) and a proper JSON object. The problem lies in the semantic correctness of the request is that the server cannot process it since the entity described is impossible within the domain logic (it cannot attach a sensor to a non-existent room). The 404 error code should be used only if the resource itself doesn't exist, while the 422 will properly say, "Your request body is proper, but semantically unprocessable."


### Question 9: From a cybersecurity standpoint, explain the risks associated with exposing internal Java stack traces to external API consumers. What specific information could an attacker gather from such a trace?

-It is dangerous to allow users of the API to see internal stack traces from Java because these leaks out implementation details that allow attackers to discover and exploit your infrastructure. This includes the exact class/package name, method names, and line numbers (which helps identify exploitable code), frameworks/library versions used (which may be correlated with CVE vulnerabilities), and even potentially some details of the configuration including file paths, operating systems/usernames, and deployment architecture. Stack traces can also give away internal route endpoints, database or hostnames, service names, or other such information if it appears in exception messages, and it can accidentally contain snippets of any sensitive information such as IDs, tokens, query parameters, or payload data that happened to exist at the time of the error. An attacker will find it much easier to construct targeted exploits (such as dependency vulnerabilities or traversal guesses), or even social engineering/attacks on your API if they know what technology it runs behind the scenes.


### Question 10: Why is it advantageous to use JAX-RS filters for cross-cutting concerns like logging, rather than manually inserting Logger.info() statements inside every single re-source method?

-The use of JAX-RS filters for concerns such as logging offers several benefits. For instance, all behavior related to logging will be located in one location and applied to all requests and responses without being repeated in each method. This approach will eliminate the need to repeat code that could otherwise lead to some requests lacking the necessary logs. It will also ensure that resource classes remain concentrated on their intended purpose, making them easier to maintain. In addition, it will be easy to alter logging behavior by modifying the filter without having to go through multiple resource methods. Filters also provide a clear point in the processing of requests where logging can be performed. Thus, logging can occur in all cases, including failed requests due to validation or exception handling.
